# 微服务架构优化评估报告

## 1. 现状分析 


### 1.1 目录结构
当前项目目录结构如下：
- `backend/`: 核心业务服务（用户、权限、认证、日志管理）。
- `services/`:
  - `user-service/`: 仅包含配置文件的空壳服务。
  - `log-service/`: 独立的日志消费服务。
  - `agent-service/`: Python 智能体服务。
- `frontend/`: 前端项目。

**存在的问题：**
1.  **结构不统一**：核心服务 `backend` 位于根目录，而其他服务位于 `services/` 目录下，缺乏统一的微服务项目结构。
2.  **空壳服务**：`services/user-service` 缺乏实际代码，与 `backend` 功能重叠且未被使用。
3.  **命名模糊**：`backend` 命名过于通用，无法准确反映其作为“管理后台服务”或“用户中心服务”的职责。

### 1.2 功能重叠与冲突
通过代码分析发现，`backend` 和 `services/log-service` 存在严重的功能重叠和不一致：
- **日志消费**：两者都包含 Kafka 消费者 (`LogConsumer`)，用于将日志保存到 Elasticsearch。
- **实现不一致**：
  - `backend`: 监听 `sys-log-topic`，支持批量消费 (`List<ConsumerRecord>`)。
  - `log-service`: 监听 `sys-log`，仅支持单条消费 (`String`)。
- **数据模型不一致** (`SysLogDocument`)：
  - `backend`: 包含 `traceId`, `params`, `result`, `errorMsg`, `costTime` 等详细字段；IP 字段名为 `ip`；时间类型为 `Date`。
  - `log-service`: 缺少上述详细字段；IP 字段名为 `ipAddress`；时间类型为 `LocalDateTime`。
- **潜在风险**：如果直接启用 `log-service`，因字段定义和 Topic 名称不匹配，将导致**消息丢失、反序列化失败或数据写入不完整**。

### 1.3 依赖管理
- 各服务独立管理 `pom.xml` 或 `requirements.txt`，缺乏统一的版本管理（如 Maven Parent POM），容易导致依赖冲突（如 Spring Boot 版本不一致）。

### 1.4 智能体服务 (Agent Service) 现状
- **代码实现**：目前 `agent-service` 仅为骨架代码，虽然配置了 Kafka 和 Nacos 地址，但**实际代码中并未实现 Kafka 消息发送或 Nacos 注册逻辑**。
- **功能缺失**：对话接口仅返回静态或简单的回显响应，未接入实际的大模型或日志系统。

## 2. 优化建议

### 2.1 目录结构重构
建议采用标准的微服务目录结构，将所有服务统一管理：

```text
/
├── services/
│   ├── admin-service/      # 原 backend，负责用户、权限、认证及业务逻辑
│   ├── log-service/        # 负责日志收集与存储
│   └── agent-service/      # 负责 AI 智能体
├── common/                 # (可选) 公共模块，如实体类、工具类
├── frontend/               # 前端项目
├── infrastructure/         # 基础设施配置 (docker-compose, nginx, etc.)
└── doc/                    # 文档
```

**具体行动：**
1.  将 `backend` 移动到 `services/admin-service`。
2.  删除 `services/user-service`（因其为空壳且功能已在 `admin-service` 中实现）。

### 2.2 职责划分与代码调整

#### Admin Service (原 Backend)
- **职责**：用户管理、权限控制 (RBAC)、认证 (Auth)、业务逻辑、**日志查询**（提供 API 给前端展示日志）。
- **调整**：
  - **移除** `LogConsumer`：不再负责从 Kafka 消费日志。
  - **保留** `LogController` 和 `SysLogRepository`：用于查询 ES 中的日志数据。
  - **保留** `LogAspect`：负责生产日志并发送到 Kafka。
  - **配置改造**：目前 `application.yml` 中大量硬编码 `localhost`，必须改造为支持环境变量注入（推荐）或 Nacos 配置中心。

#### Log Service
- **职责**：**日志收集**（从 Kafka 消费日志并写入 ES）。
- **调整**：
  - **迁移代码**：将 `backend` 中优秀的批量消费逻辑 (`Batch Listener`) 迁移到 `log-service`。
  - **统一 Topic**：修改监听 Topic 为 `sys-log-topic`。
  - **统一数据模型**：**必须**将 `backend` 中的 `SysLogDocument` 类完全复制到 `log-service`，覆盖现有定义，确保字段名、类型完全一致（特别是 `ip` vs `ipAddress`, `Date` vs `LocalDateTime` 以及缺失的 `params`, `result` 等字段）。
  - **Nacos 依赖**：虽然配置了 Nacos，但考虑到系统规模较小，建议**暂时移除 Nacos 依赖**，改用环境变量直接配置 Kafka 和 ES 地址，以降低部署复杂度。

#### Agent Service
- **职责**：AI 智能体交互。
- **调整**：
  - **完善功能**：需补充 Kafka 生产者代码，以便将对话日志发送到 `sys-log-topic`。
  - **配置简化**：同样建议移除未使用的 Nacos 配置，改用环境变量管理配置。

### 2.3 依赖管理优化
- 建议创建一个根目录下的 `pom.xml` 作为父工程，管理 Spring Boot 版本和公共依赖版本（如 Hutool, MyBatis Plus 等），确保所有 Java 服务依赖一致。

## 3. 容器化部署架构评估

### 3.1 容器规划
根据需求，系统将包含以下容器：

**业务服务容器：**
1.  `admin-service`: 核心业务 API。
2.  `log-service`: 日志消费服务。
3.  `agent-service`: 智能体服务。
4.  `frontend`: 前端静态资源 + Nginx 反向代理。

**基础组件容器：**
1.  `mysql`: 业务数据存储。
2.  `redis`: 缓存与 Session。
3.  `elasticsearch`: 日志存储。
4.  `kafka` + `zookeeper`: 消息队列。
5.  `nacos`: (可选) 注册中心与配置中心。**建议暂不部署**，除非确实需要动态配置管理，否则徒增资源消耗和运维复杂度。

### 3.2 Nginx 部署策略建议
**建议方案：合并部署 (Frontend + Nginx)**

将 Nginx 作为前端容器的一部分，既负责托管静态资源，又负责 API 反向代理。

**理由：**
1.  **简化架构**：前端项目（React/Vue）构建后为静态文件，必须依赖 Web 服务器（如 Nginx）运行。既然前端容器必须包含 Nginx，直接利用该 Nginx 进行 API 反向代理可以减少一个独立的网关容器，降低维护成本。
2.  **配置内聚**：前端路由与 API 转发规则紧密相关（解决跨域、路径重写等），将 `nginx.conf` 放在前端项目中管理，便于前端开发人员维护。
3.  **性能**：减少了一层容器间的网络跳转（相比于 Browser -> 独立 Nginx -> 前端 Nginx -> 静态文件）。

**实施方式：**
- 在 `frontend` 目录下创建 `Dockerfile`，基于 `nginx:alpine` 镜像。
- 将构建好的 `dist` 目录复制到 `/usr/share/nginx/html`。
- 将 `infrastructure/nginx/nginx.conf` (或前端专用的配置) 复制到 `/etc/nginx/conf.d/default.conf`。
- 该容器对外暴露 80 端口，作为系统的统一入口。

### 3.3 关键依赖与启动顺序
1.  **配置管理**：
    - 鉴于 `agent-service` 未实现 Nacos 逻辑且 `admin-service` 未接入 Nacos，建议**全线采用环境变量注入**的方式进行配置管理，放弃 Nacos 以简化架构。
2.  **启动顺序**：
    - 第一梯队：MySQL, Redis, Elasticsearch, Zookeeper。
    - 第二梯队：Kafka (依赖 Zookeeper)。
    - 第三梯队：业务服务 (依赖上述所有组件)。

### 3.4 资源与网络规划
- **网络**：所有容器应加入同一 Docker Network (e.g., `trae-net`)。
- **端口映射**：
  - 前端容器 (Frontend + Nginx): 80/443 -> 宿主机。
  - 其他数据库/中间件端口建议仅在容器网络内部开放，不映射到宿主机，以增强安全性（开发环境除外）。

## 4. 本地 CI/CD 实施建议

### 4.1 工具选择
**推荐方案：Gitea + Gitea Actions**

**理由：**
1.  **轻量级**：相比 Jenkins 和 GitLab，Gitea 资源占用极低，非常适合本地开发环境。
2.  **一体化**：集成了代码托管和 CI/CD 功能，无需维护多套系统。
3.  **兼容性**：Gitea Actions 语法兼容 GitHub Actions，学习成本低，且方便未来迁移到 GitHub。

### 4.2 流水线设计
建议采用单分支（`main`）触发构建的简单流程：

1.  **代码检出 (Checkout)**：拉取最新代码。
2.  **环境准备 (Setup)**：安装 JDK 17 和 Docker Buildx。
3.  **构建与测试 (Build & Test)**：
    - `admin-service`: `mvn clean package`
    - `log-service`: `mvn clean package`
4.  **镜像构建 (Docker Build)**：使用 `docker-compose build` 构建所有服务镜像。
5.  **部署 (Deploy)**：使用 `docker-compose up -d` 更新本地容器。

### 4.3 配置文件示例
在项目根目录创建 `.gitea/workflows/build.yml`：

```yaml
name: Build and Deploy

on:
  push:
    branches:
      - main

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Set up JDK 17
        uses: actions/setup-java@v3
        with:
          java-version: '17'
          distribution: 'temurin'
          cache: maven

      - name: Build Admin Service
        run: |
          cd services/admin-service
          mvn clean package -DskipTests

      - name: Build Log Service
        run: |
          cd services/log-service
          mvn clean package -DskipTests

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v2

      - name: Build and Push Docker Images
        run: |
          docker-compose build

      - name: Deploy
        run: |
          docker-compose up -d
```

## 5. 实施路线图

1.  **清理与重命名**：
    - 删除 `services/user-service`。
    - 重命名 `backend` 为 `services/admin-service`。
2.  **代码解耦与修正**：
    - **关键步骤**：将 `backend` 的 `SysLogDocument.java` 复制到 `log-service`，解决实体类不一致问题。
    - 从 `admin-service` 中删除 `LogConsumer`。
    - 更新 `log-service` 的 `LogConsumer` 以支持批量消费，并统一 Topic 为 `sys-log-topic`。
    - **移除 Nacos 依赖**：修改 `log-service` 和 `agent-service` 配置，移除 Nacos 相关依赖，改用纯环境变量配置。
3.  **配置统一**：
    - 改造 `admin-service` 的 `application.yml`，支持环境变量。
    - 确保所有服务使用服务名（如 `mysql`, `redis`）而非 `localhost` 连接组件。
4.  **容器化**：
    - 编写 `docker-compose.yml`，定义所有服务和网络。
    - **前端容器化**：编写 `frontend/Dockerfile`，集成 Nginx 配置，实现静态托管与 API 代理一体化。
5.  **CI/CD 搭建**：
    - 部署 Gitea 服务。
    - 配置 Runner。
    - 提交 `.gitea/workflows/build.yml`。

## 6. 结论
当前架构存在结构混乱和职责不清的问题，特别是日志处理逻辑分散在两个服务中，且存在**严重的数据模型不一致**。必须优先解决实体类定义差异和 Topic 名称不一致的问题。此外，鉴于 `agent-service` 尚未真正集成 Nacos 且系统规模较小，建议**移除 Nacos**，采用更轻量级的环境变量配置方案，以降低系统复杂度和资源消耗。通过引入 Gitea Actions，可以低成本实现本地自动化构建与部署，提升开发效率。

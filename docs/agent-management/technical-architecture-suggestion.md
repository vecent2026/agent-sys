# Agent 管理平台技术架构设计建议方案（前瞻版）

## 1. 文档说明

- 文档类型：技术架构设计（前瞻性）
- 定位：结合智能体领域技术发展趋势，给出具备演进空间的平台架构建议
- 版本：v1.0

> 本文档不拘泥于当前技术栈约束，以"能力建设"为导向，以"演进性"为原则，适度参考业界主流方案（LangGraph、AutoGen、Dify、MCP 协议、A2A 协议等）给出架构建议。字段级设计以后续详设文档承接。

## 2. 技术趋势判断

### 2.1 行业关键变化

过去一年，智能体领域出现了几个影响架构选型的关键趋势：

| 趋势 | 代表性进展 | 对平台的影响 |
|------|-----------|-------------|
| **多智能体协作成为主流范式** | AutoGen v0.4、LangGraph、CrewAI | 平台需从"单体智能体"转向"多 Agent 编排能力" |
| **工具集成标准化：MCP 协议** | Anthropic 发布 MCP，Claude / OpenAI / Cursor 等主流产品跟进 | 工具接入应以 MCP 为标准，构建开放工具生态 |
| **Agent 间通信标准化：A2A 协议** | Google 联合主流厂商发布 Agent-to-Agent Protocol | 多智能体间调用应考虑 A2A 协议兼容性 |
| **LLM 网关成为基础设施** | LiteLLM、PortKey、One API 等产品成熟 | 模型接入层应解耦为独立 LLM Gateway 服务 |
| **Agentic RAG 替代传统 RAG** | GraphRAG、HyDE、Self-RAG 等技术落地 | 知识能力应设计为独立可插拔的 Retrieval Service |
| **Agent 可观测性专业化** | LangSmith、Phoenix、Arize 等平台兴起 | Trace 需从 LLM 级延伸到 Agent Plan 级、工具调用级 |
| **推理模型（Reasoning Model）崛起** | o3、Gemini 2.0 Flash Thinking、DeepSeek-R2 | 模型层需支持推理模型的差异化参数与流式 thinking 输出 |
| **Human-in-the-loop 成为必要能力** | 高风险场景需人工节点审批 | 工作流引擎需原生支持人工干预节点 |

### 2.2 核心设计原则

基于以上判断，平台架构坚持以下原则：

1. **能力解耦**：LLM 接入、工具调用、记忆、知识检索各自独立，可单独演进
2. **协议优先**：工具层遵循 MCP 协议，Agent 间调用预留 A2A 兼容层
3. **运行时可插拔**：支持切换不同 Agent 执行引擎（内置 / LangGraph / AutoGen）
4. **全链路可观测**：Trace 粒度到每一步推理、工具调用、子 Agent 调用
5. **演进式架构**：MVP 阶段聚焦单智能体，架构设计为多智能体预留扩展点

## 4. 核心能力模块设计

### 4.1 LLM Gateway（模型网关）

#### 4.1.1 定位

模型网关是独立的基础设施服务，**不绑定** Agent 业务逻辑。它的职责是：统一模型访问入口、隔离厂商差异、提供可靠性保障与成本管控。

**推荐参考：LiteLLM / PortKey 的设计思路**，如果团队规模允许，可直接引入 LiteLLM 作为网关，而不是自研。

#### 4.1.2 核心能力

```
LLM Gateway
├── 模型路由
│   ├── 按模型标识路由到对应厂商适配器
│   ├── 负载均衡（同一模型多个 Key 轮转）
│   └── 故障转移（主模型不可用时自动切换备用模型）
│
├── 可靠性保障
│   ├── 重试策略（指数退避）
│   ├── 限流（全局 TPM / RPM 配额管控）
│   ├── 熔断（连续失败自动断路，降级到备用）
│   └── 超时控制
│
├── 成本管控
│   ├── Token 计量（输入/输出分别统计）
│   ├── 按调用方（agentId）分摊成本
│   └── 预算告警
│
└── 统一接口规范
    ├── Chat Completions（兼容 OpenAI 格式）
    ├── 流式响应（SSE）
    ├── 推理模型支持（thinking token 透传）
    └── Embedding（知识服务所需）
```

#### 4.1.3 支持的模型类型

| 类型 | 代表 | 特殊处理 |
|------|------|---------|
| 通用对话模型 | GPT-4o、Claude 3.5、通义 | 标准 Chat API |
| 推理/思维链模型 | o3、DeepSeek-R1、Gemini Thinking | 支持 thinking token 流式透传、预算 token 参数 |
| 多模态模型 | GPT-4V、Claude 3.5 Sonnet | 图片 / 文件输入支持 |
| 本地部署模型 | Ollama / vLLM | 自建网关适配 |
| Embedding 模型 | text-embedding-3、BGE | 知识服务专用 |

### 4.2 Agent Studio（配置管理台）

#### 4.2.1 智能体的核心构成

一个智能体不再只是"提示词 + 模型"，现代智能体的配置维度更为丰富：

```
Agent 配置
├── 身份定义
│   ├── 系统提示词（支持模板变量）
│   ├── 角色与能力描述（用于多 Agent 场景中的自我介绍）
│   └── 输出格式（文本 / JSON Schema 约束的结构化输出）
│
├── 模型配置
│   ├── 主模型选择（从 LLM Gateway 可用列表选择）
│   ├── 推理参数（temperature、top_p、max_tokens）
│   ├── 推理预算（针对推理模型的 thinking budget）
│   └── 备用模型（fallback，故障时自动切换）
│
├── 工具配置（MCP-based）
│   ├── 内置工具（搜索、代码执行、计算器等）
│   ├── 外部 MCP Server 接入（HTTP / stdio）
│   └── 工具调用策略（auto / required / none）
│
├── 记忆配置
│   ├── 上下文窗口策略（截断 / 摘要 / 滑动窗口）
│   ├── 会话记忆（跨对话持久化关键信息）
│   └── 用户级个性化记忆开关
│
├── 知识配置
│   ├── 关联知识库列表
│   ├── 检索策略（向量检索 / 关键词 / 混合）
│   └── 检索结果注入位置（system / user）
│
└── 运行策略
    ├── 最大循环步数（防止无限推理）
    ├── Human-in-the-loop 节点（需审批的操作类型）
    └── 超时设置（整体任务级）
```

#### 4.2.2 提示词管理

提示词应作为独立能力建设，而非只是 Agent 配置中的一个字段：

- **模板化**：支持 Jinja2 / Mustache 风格变量插值，引用上下文变量（用户信息、日期、业务数据）
- **版本化**：提示词独立版本管理，可与智能体版本解耦迭代
- **A/B 测试**：同一智能体可挂载多个提示词版本，按比例灰度，结合评估系统自动收敛到最优版本
- **提示词库 → Skills 升级路径**：单个 Agent 内的提示词模板，当发现在多个场景中重复使用时，应将其"晋升"为 Skill Library 中的独立 Skill（见 4.4 节），实现平台级复用而非各自维护副本

### 4.3 Agent Runtime（执行引擎）

#### 4.3.1 执行范式

不同业务场景对 Agent 执行方式的需求差异较大，运行时需支持多种范式：

| 范式 | 适用场景 | 代表框架 |
|------|---------|---------|
| **ReAct**（推理 + 行动循环） | 工具调用、信息检索、问答 | LangChain Agent |
| **Plan-and-Execute** | 复杂多步任务、代码生成 | LangGraph |
| **Reflection** | 写作、内容优化，需要自我批评 | Reflexion |
| **直接响应（Direct）** | 简单问答、格式转换 | 无需循环 |

执行引擎应以**插件化方式**支持多种范式，智能体配置时选择执行范式，平台统一调度。

#### 4.3.2 执行引擎架构

```
Agent Runtime
│
├── Session Manager（会话管理）
│   ├── 管理会话上下文生命周期
│   ├── 多轮对话状态维护
│   └── 流式响应控制（SSE）
│
├── Plan Engine（规划引擎）
│   ├── 将用户意图分解为执行步骤
│   ├── 步骤依赖分析（串行 / 并行）
│   └── 动态计划调整（执行中重规划）
│
├── Step Executor（步骤执行器）
│   ├── LLM 调用（通过 LLM Gateway）
│   ├── Tool 调用（通过 Tool Service / MCP）
│   ├── Sub-Agent 调用（通过 Orchestrator）
│   └── 人工节点（暂停等待审批）
│
├── State Manager（状态管理）
│   ├── 执行状态持久化（支持断点续跑）
│   ├── 中间结果存储
│   └── 错误状态记录
│
└── Tracer（链路追踪）
    └── 每步操作上报 Trace（与 Observability 集成）
```

#### 4.3.3 关于引擎选型的建议

> 自研执行引擎成本较高，建议综合评估以下路径：

- **路径 A（自研）**：完全控制，适合有强定制需求、且团队有 AI 基础设施积累的情况
- **路径 B（集成 LangGraph）**：LangGraph 已成为业界主流 Agent 编排框架，状态机模型成熟，支持人工节点、持久化。建议优先评估
- **路径 C（集成 Dify 引擎）**：Dify 开源，覆盖配置 + 运行时 + 观测的完整链路，适合快速启动、团队 AI 背景较弱的情况

### 4.4 Skill Library（技能库）

#### 4.4.1 为什么需要 Skills 层

当前文档的能力层次在 Tool（代码执行）和 Agent（自主规划）之间存在一个空白：

> 很多业务任务不需要 Agent 的自主规划能力，但又不是简单的代码调用——它们需要 LLM 来处理，且应该**跨智能体复用**。

典型场景：摘要生成、实体抽取、文本分类、情绪分析、语言翻译、SQL 生成、代码解释、格式转换……

如果没有 Skills 层，这些能力的现实处境是：
- 以"系统提示词"的形式分散在各个 Agent 配置中，无法复用
- 每个 Agent 单独维护相同的提示词逻辑，迭代成本翻倍
- 在多 Agent 场景中，Orchestrator 无法感知子 Agent 的"专业能力边界"
- Workflow 的 LLM 节点缺乏标准化的能力描述，无法跨流程引用

#### 4.4.2 Skill 的定义

```
Skill = 提示词模板 + 模型配置 + 输入/输出 Schema + （可选）工具调用
```

- **确定性调用**：给定输入，行为可预期，不存在自主规划循环
- **可独立版本**：Skill 有自己的版本生命周期，独立于任何 Agent
- **可组合**：Skill 可调用 MCP Tools，也可被 Agent、Workflow 调用
- **可共享**：平台级复用，团队内任何 Agent 或 Workflow 均可引用

#### 4.4.3 Skill 类型

| 类型 | 描述 | 示例 |
|------|------|------|
| **Atomic Skill（原子技能）** | 单次 LLM 调用，一个明确任务 | 摘要、翻译、情感分类、命名实体识别 |
| **Composite Skill（复合技能）** | 多步确定性流水线（LLM + Tool 组合），无规划循环 | "研究摘要"（搜索 → 过滤 → LLM 综合）、"NL to SQL"（意图解析 → SQL 生成 → 语法校验）|
| **Extraction Skill（抽取技能）** | 结构化信息抽取，强制 JSON Schema 输出 | 从合同中抽取关键条款、从日志中抽取错误信息 |
| **Generation Skill（生成技能）** | 基于模板和上下文生成内容 | 邮件起草、代码注释生成、报告生成 |

#### 4.4.4 Skill Library 核心能力

```
Skill Library
│
├── Skill 注册与管理
│   ├── Skill 定义（提示词模板 + 参数 Schema + 模型配置）
│   ├── 版本管理（独立于 Agent 版本，可灰度发布）
│   ├── 可见性控制（平台公开 / 团队内共享 / Agent 私有）
│   └── 依赖声明（Skill 所需的 Tool 或知识库列表）
│
├── Skill 调用接口
│   ├── 直接调用（业务系统通过 API 直接调用 Skill，无需经过 Agent）
│   ├── Agent 调用（Agent 将 Skill 作为能力项注册，在推理中按需触发）
│   └── Workflow 调用（Skill 作为 Workflow 的 LLM 节点被编排）
│
├── Skill 发现与推荐
│   ├── 按能力类型检索（生成 / 抽取 / 分类 / 转换）
│   ├── 按领域检索（金融 / 法律 / 代码 / 通用）
│   └── 相似 Skill 推荐（避免重复创建）
│
└── Skill 评估与监控
    ├── 调用量、延迟、成功率统计
    ├── 基于 LLM-as-Judge 的输出质量自动评分
    └── 跨 Agent 的 Skill 效果对比
```

#### 4.4.5 Skills 在多 Agent 场景中的作用

在多 Agent 协作中，Skills 是子 Agent "专业能力"的标准化描述方式：

```
Orchestrator Agent 的任务分配逻辑：

用户任务：分析这份合同，提取关键条款并翻译为英文

Orchestrator：
  1. 调用 ContractExtractionSkill → 获得结构化条款数据
  2. 调用 TranslationSkill（target=EN） → 获得英文版本
  3. 若内容复杂 → 委派给 LegalReviewAgent（该 Agent 注册了 LegalAnalysisSkill）
  4. 汇总输出

优势：Orchestrator 按"技能"而非"Agent 名称"分配任务，
      更换底层 Agent 实现不影响编排逻辑。
```

#### 4.4.6 Skill 与其他概念的边界

| 概念 | 执行方式 | LLM 参与 | 规划循环 | 复用粒度 |
|------|---------|---------|---------|---------|
| MCP Tool | 代码执行 | 否 | 否 | 工具级 |
| **Skill** | **LLM 调用（确定性）** | **是** | **否** | **能力级（跨 Agent）** |
| Agent | LLM + 规划循环 | 是 | 是 | 实体级 |
| Workflow | 有向编排 | 可选 | 否 | 流程级 |

> **设计原则**：如果一个 LLM 处理逻辑会在超过 2 个 Agent 或场景中被用到，它就应该被封装为 Skill，而不是重复写在每个 Agent 的提示词里。

### 4.5 Multi-Agent Orchestrator（多智能体编排）

#### 4.5.1 为什么需要多 Agent

单一 Agent 在复杂任务中面临上下文窗口限制、专业领域能力天花板、并行效率低等问题。多 Agent 通过分工协作解决这些问题：

```
典型多 Agent 拓扑（结合 Skills 的能力声明）

Orchestrator Agent（规划协调）
     ├── Research Agent
     │    └── 注册 Skills：WebSearchSkill、SummarizationSkill
     ├── Code Agent
     │    └── 注册 Skills：CodeGenerationSkill、CodeReviewSkill
     ├── QA Agent
     │    └── 注册 Skills：FactCheckSkill、HallucinationDetectionSkill
     └── Summary Agent
          └── 注册 Skills：ReportGenerationSkill、TranslationSkill
```

> Orchestrator 通过子 Agent 的 **Skills 声明**决定任务分配，而不是按 Agent 名称硬编码，新增子 Agent 只需注册其 Skill 即可被 Orchestrator 感知。

#### 4.5.2 编排模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **Supervisor 模式** | Orchestrator 动态决定调用哪个子 Agent，结果汇总后再决策 | 开放性复杂任务 |
| **Pipeline 模式** | Agent 链式调用，前一个的输出作为后一个的输入 | 确定步骤的流程任务 |
| **Parallel 模式** | 多个 Agent 并发执行，结果聚合 | 独立子任务并发 |
| **Debate 模式** | 多个 Agent 对同一问题给出不同视角，最终 Aggregator 仲裁 | 高风险决策、内容审核 |

#### 4.5.3 Agent 通信设计

- **同平台内部**：通过平台内部调用（函数调用或消息队列）
- **跨平台 / 跨系统**：预留 **A2A 协议**（Agent-to-Agent Protocol）兼容层，支持标准化的 Agent 能力描述（Agent Card）与跨系统调用
- **异步 Agent**：长任务支持异步执行 + 结果回调，不阻塞调用方

### 4.6 Tool Service（工具服务 / MCP 集成）

#### 4.6.1 以 MCP 为核心的工具生态

MCP（Model Context Protocol）正在成为工具集成的事实标准。工具服务应以 MCP 为核心构建：

```
Tool Service
│
├── 内置 MCP Server（平台自带工具）
│   ├── Web Search（网络搜索）
│   ├── Code Interpreter（代码执行沙箱）
│   ├── File Reader（文档解析：PDF / Excel / Word）
│   ├── HTTP Client（外部 API 调用）
│   └── Database Query（结构化数据查询）
│
├── 外部 MCP Server 接入
│   ├── HTTP Server 模式（远程 MCP Server）
│   ├── stdio 模式（本地进程）
│   └── MCP Server 注册表（统一管理已接入的 MCP Server）
│
└── 工具调用代理
    ├── 参数校验（JSON Schema 验证）
    ├── 权限校验（哪些 Agent 可以调用哪些工具）
    ├── 执行结果格式化（统一返回给 Agent Runtime）
    └── 调用日志（工具级 Trace）
```

#### 4.6.2 工具市场（Tool Marketplace）

随着工具数量增长，应建立工具市场机制：

- 工具以 MCP Server 为单位注册，描述能力、参数规范、调用方式
- 支持公开工具（平台内所有 Agent 可用）和私有工具（特定 Agent 专用）
- 工具接入审核流程（安全扫描 + 能力测试）

### 4.7 Memory Service（记忆服务）

#### 4.7.1 三层记忆模型

现代 Agent 的记忆体系应分为三层，各层职责不同：

```
┌─────────────────────────────────────────────┐
│  Working Memory（工作记忆）                  │
│  当前会话上下文，存于 Context Window          │
│  生命周期：单次对话                          │
└─────────────────────────┬───────────────────┘
                          │ 重要信息提取
┌─────────────────────────▼───────────────────┐
│  Episodic Memory（情节记忆）                 │
│  历史会话摘要、用户偏好、交互模式             │
│  存储：向量数据库 + 结构化 DB               │
│  生命周期：用户级持久化                      │
└─────────────────────────┬───────────────────┘
                          │
┌─────────────────────────▼───────────────────┐
│  Semantic Memory（语义记忆）                 │
│  领域知识、公司文档、产品资料                │
│  存储：向量数据库（知识库）                  │
│  生命周期：知识库更新周期                    │
└─────────────────────────────────────────────┘
```

#### 4.6.2 上下文窗口管理策略

随着对话变长，上下文溢出是常见问题，应支持可配置的管理策略：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **截断（Truncation）** | 丢弃最早的消息 | 短任务、低记忆需求 |
| **摘要压缩（Summarization）** | 用 LLM 对早期对话做摘要替换 | 长对话、需保留语义 |
| **滑动窗口** | 保留最近 N 轮 + 关键信息提取 | 均衡方案 |
| **记忆提取（Memory Extraction）** | 自动识别并存入 Episodic Memory | 需要跨会话记忆的场景 |

### 4.8 Knowledge Service（知识服务）

#### 4.8.1 Agentic RAG 而非传统 RAG

传统 RAG 是"检索 → 直接回答"的单步流程，Agentic RAG 让 Agent 参与检索过程：

```
传统 RAG：
用户问题 → Embedding 检索 → 拼入 Prompt → 回答

Agentic RAG：
用户问题 → Agent 分析问题 → 主动决定是否检索、检索什么
         → 评估检索结果质量 → 必要时二次检索或追问
         → 综合多次检索结果 → 回答
```

#### 4.8.2 知识服务核心能力

```
Knowledge Service
│
├── 知识库管理
│   ├── 文档导入（PDF、Word、Markdown、网页、数据库）
│   ├── 文档解析与切片（语义切块策略）
│   └── 知识库版本管理
│
├── 索引构建
│   ├── 向量化（Embedding，支持多种 Embedding 模型）
│   ├── 关键词索引（全文检索）
│   └── 图谱索引（GraphRAG，实体关系提取）
│
├── 检索策略
│   ├── 向量语义检索
│   ├── BM25 关键词检索
│   ├── 混合检索（Hybrid Search）+ 重排序（Reranker）
│   └── 图谱增强检索（复杂关系推理场景）
│
└── 检索质量评估
    ├── 命中率、MRR、NDCG 等指标追踪
    └── 对话级检索效果反馈闭环
```

### 4.9 Workflow Engine（工作流引擎）

#### 4.9.1 为什么需要独立工作流引擎

并非所有场景都适合"完全自主的 Agent"——在企业场景中，很多流程需要确定性、可审计、可干预：

```
完全自主 Agent ←──────────────────────→ 确定性工作流
    │                                        │
适合：探索性任务                        适合：合规流程
     开放问答                                审批流程
     创意生成                                数据处理管道
```

工作流引擎让平台覆盖确定性场景，与 Agent Runtime 协同。

#### 4.9.2 工作流核心节点类型

```
Workflow Node Types
│
├── 触发节点：手动触发 / Webhook / 定时任务 / 事件驱动
├── Skill 节点：调用 Skill Library 中的技能（推荐替代裸 LLM 节点）
├── LLM 节点：调用单次 LLM 生成（临时/一次性，不复用时使用）
├── Agent 节点：调用一个完整的 Agent（嵌套使用）
├── Tool 节点：直接调用工具（不经过 Agent 推理）
├── 条件分支：基于变量 / LLM 判断的 if-else 分支
├── 循环节点：批量处理、列表迭代
├── 并行节点：多分支并发执行 + 结果聚合
├── 人工节点：暂停流程，等待人工审批 / 输入
├── 代码节点：执行自定义代码（数据转换、计算）
└── 子流程节点：调用另一个已发布的工作流
```

#### 4.9.3 Human-in-the-Loop 设计要点

高风险操作（如发送邮件、写入数据库、调用外部支付接口）需要人工节点：

- 工作流执行到人工节点时，状态持久化并暂停
- 向指定人员推送待审任务（集成内部通知系统）
- 审批者查看执行上下文 + Agent 推理过程后决策（通过 / 拒绝 / 修改）
- 平台记录审批决策，工作流继续或终止

### 4.10 Observability Platform（可观测平台）

#### 4.10.1 Agent Trace 的特殊性

Agent 的可观测需求远超传统服务：不只是请求/响应，还需要看到每一步推理、每一次工具调用、每一个子 Agent 的行为。

```
Trace 层次结构（树形）

Session（会话）
└── Turn（一轮对话）
    └── Agent Run（一次 Agent 执行）
        ├── Plan（规划步骤）
        ├── Step 1: LLM Call
        │   ├── Input: messages
        │   ├── Output: tool_call / text
        │   ├── tokens: prompt=1200, completion=80
        │   └── latency: 1.2s
        ├── Step 2: Tool Call（web_search）
        │   ├── Input: {"query": "..."}
        │   ├── Output: search results
        │   └── latency: 0.8s
        ├── Step 3: LLM Call（综合结果）
        └── Final Output
```

#### 4.10.2 可观测能力矩阵

| 维度 | 能力 | 用途 |
|------|------|------|
| **Tracing** | 全链路 Span 采集（兼容 OpenTelemetry） | 问题排查、性能分析 |
| **Metrics** | 调用量、成功率、延迟分布、Token 消耗 | 监控告警、容量规划 |
| **Logging** | 结构化日志，关联 Trace ID | 详细错误排查 |
| **成本视图** | 按 Agent / 用户 / 时间段的 Token 成本统计 | 成本管控、预算分配 |
| **会话回放** | 完整对话 + 推理过程可视化回放 | 效果复盘、问题定位 |
| **异常分类** | 失败原因自动分类（超时 / 工具失败 / 模型拒绝 / 幻觉等） | 快速定位根因 |

#### 4.10.3 评估体系（Evaluation）

可观测的终极目标是支撑**持续改进**，需要建立评估闭环：

```
评估闭环
│
├── 自动评估（无需标注）
│   ├── 格式合规性（JSON Schema 校验）
│   ├── 工具调用正确率
│   └── 基于 LLM-as-Judge 的质量打分（相关性、完整性、准确性）
│
├── 人工评估（抽样标注）
│   ├── 好评 / 差评（用户反馈）
│   ├── 专家打分（内容质量）
│   └── 对比评估（多版本对比）
│
└── 评估驱动迭代
    ├── 评估指标下降自动告警
    ├── 关联差评会话 → 定位问题根因
    └── 支撑提示词 A/B 测试收敛决策
```

### 4.11 Security & Guardrails（安全与防护）

#### 4.11.1 多层防护设计

```
Input Guardrails（输入防护）
├── Prompt Injection 检测（防止用户通过输入改写 Agent 行为）
├── 敏感信息识别（身份证、手机号、密码等 PII 数据脱敏）
└── 内容安全过滤（违规内容识别）

Runtime Guardrails（运行时防护）
├── 工具调用权限边界（Agent 只能调用被授权的工具）
├── 外部请求域名白名单
└── 敏感操作人工审批节点

Output Guardrails（输出防护）
├── 幻觉检测（关键信息来源可溯源性验证）
├── 输出内容合规过滤
└── 结构化输出 Schema 校验
```

#### 4.11.2 权限边界设计

- Agent 调用工具前，Tool Service 验证该 Agent 是否有权调用该工具
- 工具调用结果中的敏感数据（如数据库原始数据）在返回给 LLM 前做脱敏处理
- 高权限操作（写入类、外发类）统一经过人工节点或二次鉴权

## 5. 数据架构建议

### 5.1 存储选型原则

| 数据类型 | 推荐存储 | 理由 |
|---------|---------|------|
| 配置数据（Agent / Model / Tool） | 关系型数据库（MySQL / PostgreSQL） | 强一致性、事务、关联查询 |
| 会话消息记录 | 文档数据库（MongoDB）或 关系型 + JSON | 消息结构灵活，读多写少 |
| 向量数据（知识库） | 向量数据库（Milvus / Qdrant / pgvector） | 高维向量 ANN 检索性能 |
| 调用日志 / Trace | Elasticsearch | 全文检索、复杂聚合分析 |
| 热点配置缓存 | Redis | 低延迟读取 |
| 异步事件流 | Kafka | Agent 调用事件异步处理 |
| 工作流执行状态 | 关系型 + Redis | 持久化 + 实时查询 |

### 5.2 数据流向

```
Agent 调用请求
    │
    ▼
Agent Runtime（实时处理）
    │
    ├──► LLM Gateway → 返回结果
    │
    ├──► Tool Service → 工具执行结果
    │
    └──► 发出 AgentCallEvent → Kafka
                                    │
                         ┌──────────┴──────────┐
                         ▼                       ▼
                    Trace Collector          Cost Aggregator
                    → Elasticsearch          → MySQL 成本表
                    （会话 + 步骤明细）        （按天/Agent 聚合）
```

## 7. 演进路线图

### Phase 1：MVP（0-3 个月）

**目标**：让业务团队能在平台上配置和发布第一个可用的智能体

- LLM Gateway：支持 3-5 种主流模型厂商，统一接口
- Agent Studio：单智能体配置（提示词 + 模型 + 基础工具）
- 基础 Tool：Web 搜索、HTTP 调用、知识库检索
- 简单 RAG：文档上传 + 向量检索
- 调试对话：草稿态对话测试

- 基础 Trace：会话级调用记录

### Phase 2：成长期（3-9 个月）

**目标**：提升智能体能力上限，引入 Skill 复用机制与多 Agent 协作

- **Skill Library**：建立平台级 Skill 注册与复用机制，将团队内高频 LLM 处理逻辑从各 Agent 配置中抽取、沉淀为可共享的 Skill
- MCP 工具市场：开放外部 MCP Server 接入
- 工作流引擎：可视化编排（含 Skill 节点）、人工审批节点
- Multi-Agent：Supervisor 模式，子 Agent 通过 Skill 声明专业能力，支持任务拆解分配
- Agentic RAG：多步检索、检索质量评估
- 多层记忆：情节记忆持久化，跨会话用户记忆
- 评估体系：LLM-as-Judge 自动评估（含 Skill 级评估）+ 用户反馈闭环
- 成本视图：精细化 Token 成本统计（可下钻到 Skill 级别）与预算管控

### Phase 3：成熟期（9-18 个月）

**目标**：构建开放的智能体生态，支撑企业级治理

- A2A 协议：跨平台 Agent 互联互通，Skill 随 Agent Card 暴露给外部 Agent
- Skill 市场：沉淀跨团队、跨业务线的 Skill，建立内部 Skill 共享生态
- Agent 市场：沉淀可复用的 Agent 模板（Agent 可携带其 Skill 集合一并发布）
- 自动优化：基于评估指标自动触发提示词 / Skill 优化建议
- 多租户：支持组织级隔离（视业务发展决策）
- 合规与审计：完整的操作审计、数据流转记录
- 边缘部署：支持敏感数据不出域的本地模型部署方案

## 8. 关键架构决策建议

### 决策 1：语言选型

| 选项 | 优势 | 劣势 |
|------|------|------|
| **Python 后端** | AI 生态最丰富（LangGraph、RAGAS、Guardrails AI 等直接集成）| 性能、类型系统不如 Java 成熟 |
| **Java 后端** | 团队熟悉、生态成熟、性能可靠 | AI 框架生态明显弱于 Python，需要大量跨语言适配 |
| **混合架构** | 核心业务逻辑（Java）+ AI 执行链路（Python 微服务）| 系统复杂度增加，需要维护两套技术栈 |

**建议**：如果团队 AI 能力建设是长期战略，强烈建议在 AI 执行链路（Agent Runtime、LLM Gateway、知识检索）使用 Python 微服务，Java 负责业务配置管理和权限控制。这样可以直接利用 LangGraph、LiteLLM 等成熟框架，避免重复造轮子。

### 决策 2：自研 vs 集成

> 核心原则：**不要在"管道"上自研，要在"内容"上投入**

| 能力 | 建议 | 理由 |
|------|------|------|
| LLM 多厂商适配 | 集成 LiteLLM | 已覆盖 100+ 模型，维护成本极低 |
| Agent 执行引擎 | 集成 LangGraph | 状态机模型成熟，社区活跃，持续演进 |
| **Skill Library** | **自研**（参考 Semantic Kernel Plugin 规范）| Skill 是平台差异化核心，需要与业务深度结合，但规范参考成熟标准 |
| 向量检索 | 集成 pgvector 或 Qdrant | 无需维护向量数据库基础设施 |
| 业务配置管理 | 自研 | 这是团队的核心差异化价值 |
| 可观测 UI | 自研（复用现有组件）| 贴合内部使用习惯 |
| Guardrails 规则 | 部分自研 + 开源结合 | 业务规则需要定制 |

### 决策 3：流式响应策略

用户对 Agent 响应速度极为敏感，流式输出应作为一等公民：

- LLM 调用全程 SSE 流式，避免长时间白屏
- 工具调用过程应实时推送状态（"正在搜索..."、"已获取 3 条结果"）
- 多 Agent 场景下，子 Agent 的推理过程可选择性透出（"子任务 2/3 完成"）
- 工作流节点切换时实时推送进度

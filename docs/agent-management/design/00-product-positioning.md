## 2. 产品定位

### 2.1 一句话描述

**面向企业内部的 SaaS 化多租户智能体服务**：平台方运营底层 LLM 基础设施，各租户在平台上自主创建、迭代、发布和运营自己的智能体，无需关心大模型接入、密钥管理等底层技术细节。

### 2.2 核心价值主张

| 角色           | 痛点                                    | 平台提供的价值                                  |
| ------------ | ------------------------------------- | ---------------------------------------- |
| **平台管理员**    | 需要统一管理多个 LLM 供应商、API Key 安全存储、模型可用性监控 | 集中式 LLM Gateway，多供应商统一接入，密钥加密管理，可用性监控与告警 |
| **租户管理员**    | 需要具备技术能力才能接入大模型；智能体配置分散；难以迭代和复盘       | 无需写代码，配置即可上线；统一的「搭建→调试→发布→观测→迭代」闭环       |
| **开发者（租户侧）** | 无标准化的智能体调用接口；多个系统各自维护类似的 LLM 调用逻辑     | 统一 REST API / SSE 流式接口，一次发布，多处调用         |

### 2.3 产品形态

```
                    ┌─────────────────────────────────────┐
                    │           平台管理层                  │
                    │   大模型接入 / 密钥管理 / 工具市场     │
                    │   平台公共 Skill / 全局安全策略        │
                    └──────────────┬──────────────────────┘
                                   │ 平台提供能力
              ┌────────────────────┼────────────────────┐
              │                    │                    │
      ┌───────▼───────┐   ┌────────▼──────┐   ┌────────▼──────┐
      │   租户 A       │   │   租户 B      │   │   租户 C      │
      │  知识库 / 工具  │   │ 知识库 / 工具  │   │ 知识库 / 工具  │
      │  Agent / Skill │   │ Agent / Skill │   │ Agent / Skill │
      │  工作流 / 观测  │   │ 工作流 / 观测  │   │ 工作流 / 观测  │
      └───────────────┘   └───────────────┘   └───────────────┘
```

**双端产品入口：**

- **平台端**（`/platform`）：平台管理员使用，管理 LLM 模型、密钥、公共工具、平台用户、安全策略
- **租户端**（`/`）：租户管理员和用户使用，创建和管理本租户的智能体、知识库、Skill、工作流

## 3. 用户角色定义

### 3.1 角色总览

```
平台层角色
├── 超级管理员（Super Admin）：平台最高权限，管理平台设置、租户创建
└── 平台管理员（Platform Admin）：管理 LLM 模型接入、密钥、公共工具

租户层角色
├── 租户管理员（Tenant Admin）：管理本租户所有资源，含智能体/知识库/工作流
├── 租户开发者（Tenant Developer）：创建和配置智能体、Skill、知识库，可发布
└── 租户普通用户（Tenant User）：（未来）使用已发布的智能体对话服务
```

### 3.2 各角色核心操作

**平台管理员核心操作：**

- 接入大模型（配置供应商信息、API Key、测试可用性）
- 管理模型可见性（哪些模型对哪些租户开放）
- 管理平台公共工具（MCP Server 级别的工具注册）
- 管理平台级 Skill 库（可跨租户使用的公共 Skill）
- 查看平台级用量与成本统计

**租户管理员核心操作：**

- 全量管理本租户资源（智能体、知识库、Skill、工作流、工具）
- 管理租户成员权限
- 查看租户级用量与成本统计
- 配置租户级安全策略

**租户开发者核心操作：**

- 创建和迭代智能体（配置、调试、发布）
- 创建和管理知识库（上传文档、配置检索策略）
- 创建 Skill（封装可复用的 LLM 处理逻辑）
- 创建工作流（编排复杂流程）
- 查看运行观测数据

### 4.3 差异化定位

本平台与竞品的核心差异在于三点：

**差异一：企业级 SaaS 多租户隔离**

- Dify/Coze 主要面向单组织内部使用，多租户能力弱
- 本平台原生多租户设计（数据隔离、权限隔离、成本隔离），适合「平台方 + 多租户」的 SaaS 化运营模式

**差异二：与现有企业管理体系深度集成**

- 继承 admin-service 已有的 RBAC 权限体系、操作日志、用户管理
- 平台层与租户层权限模型对齐，无缝融入已有管理平台

**差异三：完整的可观测性体系**

- 继承现有 Elasticsearch + Kafka 基础设施
- 全链路 Trace（会话 → 推理步骤 → 工具调用 → 子 Agent）
- 企业级的成本分析和评估体系

### 5.2 与 admin-service 的职责边界

| 能力                | 负责方           | 说明                                                   |
| ----------------- | ------------- | ---------------------------------------------------- |
| 用户身份认证（JWT 签发/验证） | admin-service | agent-service 复用同一 JWT secret 验证 token，不重复实现         |
| 租户信息查询            | admin-service | agent-service 调用 admin-service 内部接口获取租户信息            |
| 操作日志写入            | admin-service | agent-service 在关键操作后发送事件通知 admin-service 写日志         |
| RBAC 权限校验         | admin-service | agent-service 接口从 JWT 中读取 userId/tenantId，权限校验依赖现有体系 |
| 智能体领域所有能力         | agent-service | 包括 LLM 接入、知识库、Skill、工作流、调试发布、观测                      |

## 6. 技术选型说明

### 6.1 agent-service 为何选择 Python + FastAPI

| 维度         | Python + FastAPI                                          | Java + Spring Boot                  |
| ---------- | --------------------------------------------------------- | ----------------------------------- |
| AI 生态      | ★★★★★（LiteLLM / LangGraph / RAGAS / Guardrails AI 等可直接使用） | ★★（主要靠 HTTP 调用 Python 服务适配）         |
| 异步支持       | ★★★★★（原生 asyncio，天然适合 LLM 流式 + 并发 I/O）                    | ★★★★（Project Loom 后改善，但生态不如 Python） |
| Agent 框架集成 | ★★★★★（LangGraph / CrewAI / AutoGen 均原生 Python）            | ★★（需要大量 JNI 或 HTTP 桥接）              |
| 团队已有积累     | 可学习成本（参考 admin-service 风格约定）                              | 已有完整积累（admin-service 代码参考）          |
| **决策**     | **选择 Python + FastAPI**                                   | 已有 Java 体系由 admin-service 承载        |

**关键理由**：智能体是本平台的核心差异化，后续 LangGraph 编排、Agentic RAG、多 Agent 协作等能力均深度依赖 Python AI 生态。在这个核心模块使用 Python 是长期正确的架构决策。

### 6.2 密钥管理选型

| 方案               | 优点                | 缺点                     | 选择         |
| ---------------- | ----------------- | ---------------------- | ---------- |
| AES-256-GCM 自建加密 | 实现简单，无外部依赖，部署方便   | 密钥轮换需手动处理，加密 Key 管理需谨慎 | ✅ **当前选择** |
| HashiCorp Vault  | 企业级密钥管理，自动轮换，审计完整 | 引入新基础设施，运维成本高          | 预留迁移路径     |
| 云厂商 KMS          | 托管服务，安全性高         | 与特定云厂商绑定               | 预留迁移路径     |

**当前方案**：`agent_secret_config` 表中使用 AES-256-GCM 加密存储 API Key，加密 Master Key 通过环境变量注入。设计 `SecretService` 抽象接口，后续可无缝迁移到 Vault 或 KMS。

## 7. 非功能性需求

### 7.1 性能目标

| 指标                  | 目标                |
| ------------------- | ----------------- |
| 配置读取（CRUD）接口 P95 延迟 | < 100ms           |
| 智能体调用首字节时间（TTFT）    | < 3s（受 LLM 供应商影响） |
| 一键测试接口响应时间          | < 30s（含 LLM 调用超时） |
| 批量测试并发度             | 支持单租户 10 并发测试     |

### 7.2 可靠性目标

| 指标      | 目标                                 |
| ------- | ---------------------------------- |
| 服务可用性   | 99.9%（agent-service 自身，不含 LLM 供应商） |
| 配置数据持久化 | 100%（MySQL 主从）                     |
| 调用日志完整性 | > 99.5%（Kafka 异步，允许极少量丢失）          |

### 7.3 安全要求

- API Key 明文永不出现在日志、响应体、URL 中
- 租户数据严格隔离，不可跨租户查询（所有 SQL 自动注入 tenant\_id 过滤）
- 所有接口需要有效 JWT 认证
- 管理接口（模型管理）仅平台管理员可访问

## 8. 里程碑规划（高层次）

| 阶段          | 代号  | 核心目标                      | 包含能力域                                                      |
| ----------- | --- | ------------------------- | ---------------------------------------------------------- |
| **Phase 1** | MVP | 让租户能配置并发布第一个可用智能体         | LLM 接入管理 + 智能体核心配置 + 草稿调试 + 发布管理 + 基础 REST API             |
| **Phase 2** | 成长期 | 补全知识、工具、Skill 核心能力，建立评估闭环 | 知识库 RAG + 内置工具 + Skill 库 + 批量测试 + 基础可观测性                   |
| **Phase 3** | 增强期 | 提升自动化和工程化能力               | 工作流引擎 + MCP 接入 + 灰度发布 + 成本分析 + 评估体系                        |
| **Phase 4** | 成熟期 | 构建开放生态，支持复杂编排场景           | 多 Agent 编排 + A2A 协议 + Skill 市场 + 多渠道集成 + Human-in-the-Loop |

> 各阶段具体能力清单见 `01-capability-roadmap.md`。

## 1. 功能概述

### 1.1 知识库的作用

知识库（Knowledge Base）是平台提供给智能体的私有信息源，用于弥补大语言模型训练数据截止日期之后的知识盲区、企业/租户私有业务知识，以及对精确性要求极高的结构化信息。

知识库通过检索增强生成（Retrieval-Augmented Generation，RAG）技术，在用户向智能体发起对话时，自动从知识库中检索与用户问题最相关的文本片段（Chunk），并将其注入到发送给大语言模型的上下文中，从而使 LLM 能够基于真实的私有数据给出准确答案。

### 1.2 与智能体的关系

```
用户提问
   │
   ▼
智能体（Agent）
   │
   ├─── 关联知识库 1 ──► 向量检索 ──► 命中 Chunks
   ├─── 关联知识库 2 ──► 向量检索 ──► 命中 Chunks
   │
   ▼
Chunks 注入上下文 → LLM 推理 → 带引用标注的响应
```

核心关系：

- 一个智能体可关联多个知识库（多知识库并行检索后合并排序）
- 一个知识库可被同一租户下的多个智能体共享
- 平台级公共知识库可跨租户使用（需平台管理员授权）
- 知识库与智能体解耦：知识库的文档变更不影响已发布的智能体版本，但会影响检索结果（实时生效）

## 2. 知识库管理

### 2.1 创建知识库

通过列表页工具栏「新建知识库」按钮触发，右侧 Drawer 承载创建表单。

#### 字段说明

| 字段名 | 类型 | 是否必填 | 默认值 | 说明 |
|--------|------|----------|--------|------|
| name | string | 是 | — | 知识库名称，同一租户内不可重复，最长 64 字符 |
| description | string | 否 | — | 知识库描述，最长 256 字符 |
| embedding_model_id | string | 是 | 租户默认 Embedding 模型 | 选择 Embedding 模型，创建后不可修改（修改需重建索引） |
| retrieval_strategy | enum | 是 | hybrid | 默认检索策略：vector（纯向量）/ bm25（关键词）/ hybrid（混合） |
| status | enum | 是 | active | 知识库状态：active（启用）/ inactive（禁用） |
| visibility | enum | 是 | tenant | 可见范围：tenant（租户内共享）/ platform（平台公共） |

> **注意**：`embedding_model_id` 一旦确定，该知识库下所有文档的向量均以此模型生成。若需更换模型，须新建知识库并重新导入文档。

### 2.2 知识库列表页

采用「上方工具栏 + 下方表格 + 底部操作栏」三段结构。

**工具栏**（左侧展示标题与数量，右侧为操作按钮区）：

- 搜索框：按知识库名称实时过滤
- 状态筛选：全部 / 启用 / 禁用
- 「新建知识库」按钮（primary 样式）

**表格列说明**：

| 列名 | 说明 |
|------|------|
| 名称 | 知识库名称，可点击进入文档管理页 |
| 文档数 | 该知识库下已导入的文档总数 |
| 存储大小 | 所有原始文档文件大小之和（格式：自动换算为 KB/MB/GB） |
| Embedding 模型 | 创建时选择的 Embedding 模型名称 |
| 可见范围 | 租户内 / 平台公共 |
| 状态 | 启用（绿色 Badge） / 禁用（灰色 Badge） |
| 操作 | 编辑 / 文档管理 / 删除（danger 样式，二次确认） |

分页：默认每页 20 条，支持切换为 50 / 100 条。

### 2.3 编辑知识库

点击操作列「编辑」，右侧 Drawer 打开，可修改：名称、描述、默认检索策略、状态、可见范围。

**不可修改项**：Embedding 模型（灰色禁用状态，Tooltip 提示「创建后不可修改，如需更换请新建知识库」）。

### 2.4 删除知识库

删除前执行引用检查：

1. 查询当前知识库是否被任意智能体引用
2. 若存在引用：弹出确认框，列出所有引用该知识库的智能体名称，提示「删除后，上述智能体将无法从该知识库检索数据，是否继续？」，用户需在输入框中输入知识库名称以确认删除
3. 若无引用：普通二次确认弹框
4. 删除动作：标记为已删除（软删除），同步删除 Qdrant/pgvector 中对应的 Collection，7 天后物理删除原始文件

### 2.5 知识库共享

| 类型 | 说明 | 创建权限 | 使用权限 |
|------|------|----------|----------|
| 租户内知识库 | 仅当前租户内的智能体可引用 | 租户管理员 / 普通成员（由租户设置决定） | 同租户所有成员 |
| 平台公共知识库 | 所有租户均可引用 | 仅平台超级管理员 | 所有租户 |

## 3. 文档管理

### 3.1 支持的文件格式

| 分类 | 格式 | 扩展名 | 说明 |
|------|------|--------|------|
| 文档 | PDF | .pdf | 支持文字型 PDF，扫描件需开启 OCR（P1） |
| 文档 | Word | .docx / .doc | 保留段落结构，忽略图片 |
| 文档 | PowerPoint | .pptx / .ppt | 每页幻灯片内容作为独立段落处理 |
| 表格 | Excel | .xlsx / .xls | 逐行转为文本，每行为一个 Chunk 候选 |
| 表格 | CSV | .csv | 同 Excel，支持自定义分隔符 |
| 文本 | Markdown | .md / .mdx | 按标题层级切分，保留结构语义 |
| 文本 | 纯文本 | .txt | 按段落/固定大小切分 |
| 网页 | HTML | .html / .htm | 提取正文，过滤导航栏/页脚等无效区域 |
| 代码 | JSON | .json | 解析 JSON 结构后转为文本描述 |
| 代码 | 代码文件 | .py / .js / .ts / .java / .go 等 | 按函数/类级别切分（P1） |

**文件大小限制**：单文件最大 50MB；单次批量上传最多 20 个文件。

### 3.2 文档上传交互

文档管理页提供两种上传方式入口：

**拖拽上传区域**：
- 页面内有虚线边框的拖放区，提示文字「拖拽文件至此处，或点击选择文件」
- 拖入时边框高亮、背景色变浅蓝
- 不支持的格式：拖入后显示红色错误提示，文件不进入队列

**批量上传**：
- 支持 Ctrl/Cmd + 点击多选文件
- 上传前展示待上传文件列表（文件名 / 大小 / 格式图标）
- 支持从列表中移除单个文件
- 点击「开始上传」后批量提交，每个文件独立进度条

**上传进度展示**：
- 每个文件显示上传进度百分比 + 速率（KB/s）
- 上传完成后自动进入「待处理」状态，不需要手动触发处理

### 3.3 URL 爬取

点击工具栏「从 URL 导入」，弹出 Drawer，配置项如下：

| 字段名 | 类型 | 是否必填 | 默认值 | 说明 |
|--------|------|----------|--------|------|
| url | string（URL） | 是 | — | 起始 URL，须为 http/https 协议 |
| crawl_depth | integer | 是 | 1 | 递归爬取深度，0 = 仅爬取当前页，1 = 当前页 + 一级子链接，最大 3 |
| max_pages | integer | 是 | 10 | 最多爬取页面数，范围 1-100 |
| include_patterns | string[] | 否 | — | URL 白名单正则列表，仅爬取匹配的 URL |
| exclude_patterns | string[] | 否 | — | URL 黑名单正则列表，排除匹配的 URL |
| crawl_interval_ms | integer | 否 | 500 | 爬取间隔（毫秒），避免对目标站点造成压力，范围 100-5000 |

提交后，系统创建爬取任务，异步执行。爬取到的每个页面作为一个独立文档进入处理管道。

### 3.4 文档列表

文档管理页采用三段结构，表格列说明：

| 列名 | 说明 |
|------|------|
| 文件名 | 原始文件名（URL 导入时显示页面标题），可点击查看详情/预览 |
| 来源 | 上传（显示文件图标）/ URL 爬取（显示链接图标及原始 URL） |
| 大小 | 原始文件大小 |
| 状态 | 待处理 / 处理中（带进度百分比）/ 已完成 / 处理失败 |
| Chunk 数 | 处理完成后显示切分后的 Chunk 总数 |
| 创建时间 | 文件上传/爬取时间 |
| 操作 | 预览 / 重新处理 / 删除 |

**批量操作**：表格支持多选，工具栏出现「批量删除」「批量重新处理」按钮。

### 3.5 文档状态说明

| 状态 | 描述 | 允许的操作 |
|------|------|------------|
| 待处理（PENDING） | 文件已上传，等待处理队列调度 | 删除 |
| 处理中（PROCESSING） | 正在执行解析、切分、Embedding 管道 | — |
| 已完成（COMPLETED） | 所有 Chunk 已向量化并写入向量数据库 | 预览 / 删除 |
| 处理失败（FAILED） | 管道执行中任意步骤发生不可恢复错误 | 查看错误详情 / 重新处理 / 删除 |

处理中的文档每 3 秒自动刷新进度（前端轮询或 SSE）。

### 3.6 文档预览

点击「预览」进入文档预览面板（全屏 Drawer），包含两个 Tab：

**原文内容 Tab**：
- 展示文档原始文本（渲染 Markdown，其他格式显示纯文本）
- 右上角显示总字符数 / Token 估算数

**Chunk 列表 Tab**：
- 列表展示所有 Chunk，每个 Chunk 卡片显示：
  - Chunk 序号
  - 字符数 / Token 数
  - Chunk 文本内容（超过 200 字符折叠展示，点击展开）
  - 向量化状态（已向量化 / 向量化失败）
- 支持按关键词搜索 Chunk 内容

## 4. 文档处理管道

### 4.1 完整管道流程图

```
原始文件/URL
      │
      ▼
┌─────────────┐
│  1. 文档解析 │  ← 提取文本，识别文档结构
└─────┬───────┘
      │
      ▼
┌─────────────┐
│  2. 文本清洗 │  ← 去除噪声、格式化
└─────┬───────┘
      │
      ▼
┌─────────────┐
│  3. 文本切分 │  ← 按策略切分为 Chunks
└─────┬───────┘
      │
      ▼
┌─────────────┐
│ 4. Embedding │  ← 每个 Chunk 生成向量
└─────┬───────┘
      │
      ▼
┌─────────────────┐
│ 5. 向量存储写入  │  ← 写入 Qdrant / pgvector
└─────────────────┘
```

### 4.2 各步骤详细说明

#### 步骤 1：文档解析

根据文件格式选择对应解析器：

| 格式 | 解析器 | 说明 |
|------|--------|------|
| PDF | PyMuPDF | 提取文字层，保留段落换行 |
| Word | python-docx | 提取正文段落和表格文本 |
| Excel/CSV | pandas | 逐行读取，表头作为每行的前缀 |
| Markdown | markdown-it | 解析为 AST，保留标题层级 |
| HTML | BeautifulSoup | 提取 main/article 区域，过滤 nav/footer |
| TXT | 直接读取 | UTF-8 编码，自动检测行尾 |

解析结果输出：结构化文本段落列表（每段落保留原始位置信息，用于 Chunk 来源追溯）。

#### 步骤 2：文本清洗

- 去除多余空白行（连续 2 个以上空行合并为 1 个）
- 去除页眉页脚（PDF：基于坐标检测；HTML：已在解析时过滤）
- 去除无意义特殊字符（控制字符、零宽字符等）
- Unicode 标准化（NFC 形式）
- 去除重复内容（对相似度 >95% 的段落去重，保留第一次出现的位置）

#### 步骤 3：文本切分

详见 4.3 节。

#### 步骤 4：Embedding

- 逐批调用 Embedding 模型 API（batch_size 默认 32）
- 失败时按指数退避重试（最多 3 次）
- 每个 Chunk 对应一个 float32 向量
- 向量维度由模型决定（如 text-embedding-3-small: 1536 维）

#### 步骤 5：向量存储写入

- 写入向量数据库（默认 Qdrant）
- 每条记录存储：向量 + Payload（chunk_id / doc_id / kb_id / 原文 / 位置信息）
- 支持幂等写入（相同 chunk_id 覆盖更新）
- BM25 检索的倒排索引同步写入（基于 Qdrant 内置稀疏向量支持）

### 4.3 切分策略详细说明

知识库级别设置默认切分策略，也可在上传单个文档时覆盖。

#### 策略一：固定大小切分（Fixed Size）

| 参数 | 类型 | 默认值 | 范围 | 说明 |
|------|------|--------|------|------|
| chunk_size | integer | 512 | 100-2000 | 每个 Chunk 的最大字符数 |
| chunk_overlap | integer | 50 | 0-200 | 相邻 Chunk 的重叠字符数（须小于 chunk_size） |
| separator | string | `\n` | — | 优先在该字符处切断，避免将句子从中间截断 |

**适用场景**：内容无明显结构、对均匀切分有要求的场景（如日志文件、纯文本）。

**切分逻辑**：
1. 首先尝试在 separator 处切分
2. 若到达 separator 时已超过 chunk_size，则强制在 chunk_size 处截断
3. 下一个 Chunk 从前一个 Chunk 末尾回退 chunk_overlap 字符处开始

#### 策略二：段落切分（Paragraph）

| 参数 | 类型 | 默认值 | 范围 | 说明 |
|------|------|--------|------|------|
| max_chunk_size | integer | 1000 | 200-3000 | 单个 Chunk 上限（超过则进一步切分） |
| min_chunk_size | integer | 100 | 10-500 | 单个 Chunk 下限（小于则与相邻段落合并） |
| chunk_overlap | integer | 0 | 0-200 | 跨段落重叠字符数 |

**适用场景**：结构良好的文章、报告、文档，有自然段落分隔。

**切分逻辑**：以双换行符（`\n\n`）为段落分隔符，小于 min_chunk_size 的段落合并到下一段，超过 max_chunk_size 的段落降级为固定大小切分。

#### 策略三：标题切分（Heading-based）

| 参数 | 类型 | 默认值 | 范围 | 说明 |
|------|------|--------|------|------|
| max_heading_level | integer | 2 | 1-6 | 最深参考的标题层级（Markdown H1-H6） |
| max_chunk_size | integer | 1500 | 200-4000 | 同一标题下内容超过此值时进一步切分 |
| include_heading_in_chunk | boolean | true | — | 切分后每个 Chunk 是否保留所属标题（增强语义） |

**适用场景**：结构化技术文档、API 文档、操作手册（Markdown/Word 格式最佳）。

**切分逻辑**：解析文档标题层级结构，将每个标题节点的内容作为独立 Chunk。若 include_heading_in_chunk 为 true，则每个 Chunk 前缀其完整标题路径（如「第二章 > 2.3 配置项 > 2.3.1 基础配置」）。

#### 策略四：语义切分（Semantic）

| 参数 | 类型 | 默认值 | 范围 | 说明 |
|------|------|--------|------|------|
| similarity_threshold | float | 0.8 | 0.5-0.99 | 句子间余弦相似度低于此值时视为语义切换点 |
| max_chunk_size | integer | 1000 | 200-3000 | Chunk 大小上限 |
| sentence_window | integer | 3 | 1-10 | 计算相似度时的滑动窗口句子数 |

**适用场景**：主题多变、段落结构不规则的文档，追求最高语义连贯性。

**切分逻辑**：
1. 先将文本按句子切分（以句号/问号/感叹号为句子边界）
2. 用 Embedding 模型计算相邻句子的余弦相似度
3. 相似度低于阈值的位置标记为切分点
4. 合并切分点之间的句子，超过 max_chunk_size 时强制截断

**注意**：语义切分在处理阶段需要额外调用 Embedding 模型，处理时间较其他策略长 2-5 倍。

### 4.4 Overlap（重叠）配置说明

Overlap 机制用于缓解信息在切分边界处丢失的问题：

```
Chunk 1: [句子A  句子B  句子C  句子D]
                                ↕ overlap
Chunk 2:             [句子C  句子D  句子E  句子F]
```

配置建议：
- 对话类 / 问答类内容：overlap 设为 chunk_size 的 10%-15%
- 叙述型长文档：overlap 设为 chunk_size 的 20%-25%
- 结构化数据（表格、日志）：overlap 设为 0（避免数据行错乱）

### 4.5 Embedding 模型选项

| 模型 | 提供方 | 向量维度 | 最大输入 Token | 特点 | 适用场景 |
|------|--------|----------|----------------|------|----------|
| text-embedding-3-small | OpenAI | 1536 | 8191 | 性价比高，速度快 | 大批量文档、成本敏感场景 |
| text-embedding-3-large | OpenAI | 3072 | 8191 | 精度更高，支持 Matryoshka 降维 | 精度要求高的场景 |
| text-embedding-ada-002 | OpenAI | 1536 | 8191 | 旧版，已被 v3 系列替代 | 兼容旧系统 |
| BGE-M3 | BAAI（开源） | 1024 | 8192 | 多语言、多粒度，支持稠密+稀疏混合 | 中文为主的多语言场景 |
| M3E-base | Moka AI（开源） | 768 | 512 | 中文优化，轻量级 | 中文短文本、资源受限部署 |
| bge-large-zh-v1.5 | BAAI（开源） | 1024 | 512 | 中文语义理解能力强 | 中文长文档检索 |

部署方式：OpenAI 系列通过 API Key 调用；开源模型通过平台内置推理服务（使用 FastAPI + sentence-transformers 封装）调用。

### 4.6 向量存储选型

平台推荐使用 **Qdrant** 作为向量数据库，备选方案为 **pgvector**（适合已有 PostgreSQL 基础设施的场景）。

| 特性 | Qdrant | pgvector |
|------|--------|----------|
| 部署方式 | 独立服务（Docker） | PostgreSQL 扩展 |
| 向量检索 | HNSW 索引，高性能 | IVFFlat / HNSW |
| 稀疏向量支持 | 原生支持（BM25 混合检索必需） | 不支持（需额外倒排索引） |
| Payload 过滤 | 高性能内置过滤 | 标准 SQL WHERE |
| 水平扩展 | 原生集群模式 | 受 PostgreSQL 限制 |
| 运维复杂度 | 中等（独立服务） | 低（随 PostgreSQL 部署） |
| 推荐场景 | 生产环境、大规模知识库 | 小规模、已有 PG 基础设施 |

默认使用 Qdrant。每个知识库对应 Qdrant 中一个独立的 Collection，命名规则为 `kb_{tenant_id}_{kb_id}`。

## 5. 检索配置

### 5.1 三种检索策略详细说明

#### 策略一：向量检索（Vector Search）

**原理**：将用户查询文本转为向量，在向量数据库中通过余弦相似度（或点积）找出最近邻的 Chunk。

**特点**：
- 语义理解能力强，能识别同义词、近义表达
- 对关键词精确匹配能力弱（如专有名词、代码片段、型号编号）
- 检索速度快（HNSW 索引复杂度约 O(log n)）

**适用场景**：通用问答、语义理解要求高的场景。

#### 策略二：BM25（关键词检索）

**原理**：基于 TF-IDF 改进的词频-逆文档频率算法，统计查询词在文档中的出现频率。

**特点**：
- 对关键词、专有名词、代码精确匹配能力强
- 语义理解能力弱（「电脑」和「计算机」被视为不同词）
- 检索结果可解释性强

**适用场景**：代码文档、技术手册、包含大量专有名词的领域知识。

#### 策略三：混合检索（Hybrid Search）

**原理**：同时执行向量检索和 BM25 检索，通过 RRF（Reciprocal Rank Fusion）或加权融合合并两组结果。

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| vector_weight | float | 0.7 | 向量检索权重（与 bm25_weight 之和须为 1.0） |
| bm25_weight | float | 0.3 | BM25 检索权重 |
| fusion_method | enum | rrf | 融合方式：rrf（RRF 算法）/ weighted（加权求和） |

**特点**：综合了语义理解和关键词精确匹配两种优势，在大多数场景下效果最佳。

**推荐**：对于通用知识库，优先选择混合检索策略。

### 5.2 Reranker 配置

Reranker 在初步检索后对候选 Chunk 进行精排，提升最终返回结果的相关性。

| 参数 | 类型 | 是否必填 | 默认值 | 说明 |
|------|------|----------|--------|------|
| reranker_enabled | boolean | 是 | false | 是否启用 Reranker |
| reranker_model | string | 否 | — | Reranker 模型，启用时必填 |
| reranker_top_n | integer | 否 | 5 | Reranker 最终保留的 Chunk 数（须 ≤ top_k） |

**可选 Reranker 模型**：

| 模型 | 提供方 | 说明 |
|------|--------|------|
| bge-reranker-v2-m3 | BAAI（开源） | 多语言，综合效果好，推荐默认选项 |
| bge-reranker-large | BAAI（开源） | 中文精排效果优秀 |
| cohere-rerank-3 | Cohere | 商业 API，多语言效果顶级 |

**流程**：初步检索（top_k 个结果）→ Reranker 精排（reranker_top_n 个结果）→ 注入上下文。

### 5.3 检索参数

| 参数 | 类型 | 默认值 | 范围 | 说明 |
|------|------|--------|------|------|
| top_k | integer | 5 | 1-20 | 初步检索返回的最大 Chunk 数 |
| score_threshold | float | 0.5 | 0.0-1.0 | 相似度分数阈值，低于此值的 Chunk 不参与上下文注入（即使在 top_k 内） |
| max_chunk_chars | integer | 1000 | 100-4000 | 单个 Chunk 注入上下文时的最大字符数（超出则截断） |

参数说明：
- top_k 建议根据模型上下文窗口大小和 Chunk 平均大小综合设置
- score_threshold 过高会导致大量相关内容被过滤，过低会引入噪音；建议先通过检索测试面板调试
- max_chunk_chars 主要用于控制单次请求的 Token 消耗，避免超出模型上下文限制

### 5.4 检索结果注入方式

从知识库检索到的 Chunks 有两种注入方式：

**方式一：System Prompt 末尾注入（推荐）**

```
[System Prompt 原始内容]

---
以下是从知识库检索到的相关参考资料，请基于这些内容回答用户问题：

[Chunk 1 内容]
来源：{文档名} - 第{页码}段

[Chunk 2 内容]
来源：{文档名}
---
```

特点：对用户透明，不污染对话历史，适合大多数场景。

**方式二：用户消息前注入**

```
参考资料：
[Chunk 1 内容]
[Chunk 2 内容]

用户问题：{用户原始输入}
```

特点：强迫 LLM 在回答前先阅读参考资料，对某些不擅长处理系统提示的模型效果更好。

### 5.5 引用标注格式

开启引用标注后，智能体在响应中标注知识来源，格式如下：

**行内引用**（适合精确引用特定段落）：
```
根据公司政策，年假天数为 15 天[¹]。

---
¹ 来源：《员工手册 2025版》- 第 3 章 休假制度
```

**末尾引用列表**（适合综合多来源信息的回答）：
```
[响应正文]

**参考来源**
[1] 《产品规格书 v2.3》 - 第 2.1 节技术参数
[2] 《常见问题解答》 - 问题 15
```

## 6. Agentic RAG 设计

### 6.1 传统 RAG 与 Agentic RAG 的区别

**传统 RAG 流程**：
```
用户输入 → 直接向量检索 → 注入上下文 → LLM 生成 → 响应
```

**Agentic RAG 流程**：
```
用户输入
    │
    ▼
查询理解 & 改写
    │
    ▼
检索决策（是否需要检索？）
    ├─ 否 → 直接调用 LLM 生成 → 响应
    └─ 是
         │
         ▼
      执行检索（支持多次检索）
         │
         ▼
      结果评估（相关性是否足够？）
         ├─ 足够 → 注入上下文 → LLM 生成 → 响应
         └─ 不足 → 改写查询 → 重试（最多 N 次）
```

| 维度 | 传统 RAG | Agentic RAG |
|------|----------|-------------|
| 检索时机 | 每轮对话必检索 | 由 Agent 动态决策 |
| 查询改写 | 无 | 基于对话历史改写 |
| 多轮检索 | 不支持 | 支持，检索不满意时重试 |
| 结果评估 | 无 | 有相关性判断 |
| 推理开销 | 低 | 中（额外推理步骤） |

### 6.2 检索决策逻辑

Agent 在每轮对话开始时判断是否需要执行知识库检索：

**触发检索的条件**（任一满足）：
- 用户问题中包含知识性疑问词（「什么是」「如何」「为什么」「哪些」等）
- 用户问题涉及具体数字、日期、规格等可验证信息
- 用户问题包含知识库中可能存在的关键词
- LLM 自身不确定答案（通过 thinking 步骤判断）

**不触发检索的条件**（任一满足）：
- 纯聊天类问题（「你好」「谢谢」「再见」）
- 上轮对话已检索到足够信息，当前问题是对上轮回答的追问
- Agent 判断问题属于通用知识，知识库中不太可能有更好的答案

决策逻辑由 LLM 在 System Prompt 中的指令控制，不做硬编码规则。

### 6.3 查询改写

在多轮对话中，用户的问题往往含有指代词或省略了上文语境：

**原始对话**：
```
用户：我们公司的差旅报销标准是什么？
Agent：[回答出差补贴标准]
用户：那住宿费呢？
```

「那住宿费呢？」直接检索效果差，改写后的查询为：「公司差旅报销中住宿费的标准」。

改写机制：在构建检索 Query 时，将对话历史（最近 N 轮）与当前问题一并传给 LLM，要求其输出改写后的独立查询语句。

### 6.4 多轮检索

当首次检索结果相关度不足时，触发检索重试：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| max_retrieval_rounds | 2 | 最大检索轮数（包括首次），超过后放弃检索，使用已有结果或直接生成 |
| retry_query_rewrite | true | 重试时是否重新改写查询语句 |

重试策略：
1. 第一次检索：使用改写后的查询语句执行混合检索
2. 评估结果（见 6.5 节）
3. 若不满足阈值：扩大 top_k（×1.5），更换检索策略（如由 vector 改为 hybrid），重写查询
4. 第二次检索：使用新参数重新检索
5. 无论第二次结果如何，将已获取内容注入上下文

### 6.5 检索结果评估

评估维度：

| 评估维度 | 方法 | 通过条件 |
|----------|------|----------|
| 数量充足 | 检查命中 Chunk 数 | ≥ 1 个 Chunk（score > threshold） |
| 相关度足够 | 最高分 Chunk 的相似度分数 | 最高分 > score_threshold × 1.2 |
| 内容完整 | LLM 判断（简单推理） | LLM 认为已有足够信息回答问题 |

## 7. 知识库效果评估

### 7.1 检索测试面板

在知识库详情页提供「检索测试」Tab，允许用户在不创建智能体的情况下直接测试检索效果。

**操作流程**：
1. 在输入框输入测试 Query（如「差旅报销住宿费标准」）
2. 选择检索策略（使用知识库默认 / 临时覆盖为指定策略）
3. 配置 top_k、score_threshold
4. 点击「执行检索」
5. 结果区域展示：
   - 命中的 Chunk 列表（相关度分数 / Chunk 内容 / 来源文档 / 在文档中的位置）
   - 每个 Chunk 用彩色 Bar 直观显示相似度分数
6. 可对单个 Chunk 进行「标记为相关」/「标记为不相关」，积累标注数据

### 7.2 效果指标

知识库「数据概览」Tab 展示以下指标（按最近 7 天 / 30 天 / 自定义时间范围统计）：

| 指标 | 计算方式 | 说明 |
|------|----------|------|
| 检索命中率 | 有效检索次数 / 总检索次数 | 有效检索：score_threshold 以上至少有 1 个 Chunk |
| 平均相关度分数 | 所有命中 Chunk 最高分的均值 | 越接近 1.0 越好 |
| Top-5 召回率 | 基于人工标注数据计算（可选） | 需要先在测试面板积累标注数据 |
| 平均检索耗时 | 从收到查询到返回结果的耗时（毫秒） | 不含 Reranker 耗时 |

|----------|------|------|------|
| 创建知识库 | POST | `/api/v1/knowledge-bases` | 创建新知识库 |

| 知识库详情 | GET | `/api/v1/knowledge-bases/{kb_id}` | 获取单个知识库详情 |

| 上传文档 | POST | `/api/v1/knowledge-bases/{kb_id}/documents` | multipart/form-data，支持多文件 |
| URL 导入 | POST | `/api/v1/knowledge-bases/{kb_id}/documents/crawl` | 提交 URL 爬取任务 |

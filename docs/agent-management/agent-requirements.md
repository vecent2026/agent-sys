# Agent 管理平台需求文档（初版）

## 1. 文档信息

- 文档类型：需求文档（业务导向）
- 适用范围：`service/agent-service` 智能体管理能力建设
- 目标读者：产品、研发、测试、运营
- 版本：v0.1

## 2. 背景与目标

当前团队需要在平台内快速构建并上线智能体，用于承载问答、任务协助、业务流程自动化等场景。现阶段存在的问题主要是：

- 智能体配置分散，创建和迭代成本高。
- 大模型接入方式不统一，切换模型或供应商时成本较高。
- 缺少统一的调试、发布与效果复盘入口，难以形成稳定迭代闭环。

本项目目标是在 `agent-service` 建设统一的智能体管理能力，帮助业务团队更快完成“搭建 -> 调试 -> 发布 -> 运行 -> 复盘”的闭环。

### 2.1 业务目标

- 将智能体从“开发驱动”转为“配置驱动”，缩短上线周期。
- 支持多模型接入与统一管理，降低模型替换成本。
- 建立可观测、可评估、可迭代的运营机制，提升智能体稳定性与效果。



## 3. 约束前提（必须遵守）

### 3.1 架构约束：单租户

- 本阶段采用单租户架构，不设计多租户隔离能力。
- 所有能力默认面向同一组织内的内部使用场景。

### 3.2 服务边界约束：复用 admin-service

以下能力已在 `admin-service` 建设完成，本项目不重复建设：

- 权限管理（RBAC）
- 用户管理（后台管理用户）
- 操作日志体系（含管理侧查询能力）

`agent-service` 仅聚焦智能体领域能力，不承接通用后台治理能力。


## 4. 功能需求

### 4.1 大模型接入管理（MVP）

- 通过单一模型配置表直接管理模型列表
  - **模型管理列表功能**：  
    - 列表展示：以表格形式展示模型名称、模型标识、状态、是否默认模型、最近一次可用性测试结果状态，支持分页。  
    - 列表操作：按创建时间排序、多条件筛选与搜索（模型名称、模型标识、状态），以及行内快捷操作（一键测试、编辑、删除）。  
    - 默认模型标识：对“是否默认模型”为“是”的模型在列表中做明显标识（如徽标或标签），便于识别。  
  - **数据表字段设计**：  
    - 模型 ID（主键）  
    - 模型名称（必填，用于管理和展示）  
    - 模型标识（必填，用于实际调用时的 model 字段，如 `gpt-4.1-mini`）  
    - 厂商类型 / 渠道（如 OpenAI / Azure OpenAI / Moonshot / 通义千问 / 自建网关等，用于选择对应的后端适配器）  
    - 基础请求地址（Base URL，用于拼装实际请求地址，如 `https://api.openai.com/v1`）  
    - 接口类型 / API 路径（如 chat 接口类型或 `/v1/chat/completions` 等，用于确定具体调用路径；如厂商以“接口类型 + 版本”方式区分，则记录接口类型与 API 版本）  
    - 鉴权方式（固定为 API Key，用于统一通过请求头携带 Key 进行鉴权）  
    - 密钥配置 ID（引用统一密钥/凭据管理中的配置，不在本表直接存储明文 Key）  
    - 请求超时时间（单位：毫秒或秒，用于控制单次请求的最大等待时间）  
    - 状态（启用 / 停用）  
    - 是否默认模型（是 / 否，仅允许一个为“是”）  
    - 最近一次可用性测试结果状态（可用 / 存在问题 / 未测试）及相关信息（参考下文可用性验证）  
  - **模型增删改能力**：  
    - 新增：  
      在列表页面创建新的模型记录时，需要一次性补齐该模型对接厂商 API 所需的关键信息，确保后续可以稳定发起调用并进行一键测试。  
      - 基础信息：填写模型名称、模型标识，并选择厂商类型 / 渠道。  
      - 调用配置：配置基础请求地址（Base URL）、接口类型 / API 路径、API 版本（如适用）以及请求超时时间等基础调用参数。  
      - 鉴权配置：配置鉴权方式为 API Key，并关联密钥配置 ID（从统一密钥/凭据管理中引用，不在本表保存明文密钥）。  
      - 管理属性：设置模型状态（启用 / 停用）、是否默认模型（是 / 否，系统通过约束保证同时仅有一个默认模型）。  
      - 校验规则：提交时系统校验必填字段完整性、模型标识唯一性等规则，通过后写入模型配置表，供后续「一键测试」和智能体配置引用使用。  
    - 编辑：  
      为保证已接入模型调用的稳定性，编辑能力仅支持修改有限的管理类字段和少量安全范围内的配置，不改变其接入身份本质。  
      - 可编辑字段：模型名称、状态（启用 / 停用）、是否默认模型（将某模型设为默认时，系统自动取消其他模型的默认标记）、请求超时时间等非关键调用参数。  
      - 受限字段：厂商类型 / 渠道、模型标识、Base URL、接口类型 / API 路径、API 版本、鉴权方式、密钥配置 ID 等作为模型接入身份的核心字段，默认不支持在编辑中修改，如需变更应通过新建模型记录完成迁移。  
      - 边界说明：可用性测试结果由「一键测试」写回，不在编辑中直接修改；温度、top_p、最大输出长度等推理参数在智能体配置维度管理，不在模型管理中编辑。  
    - 删除：  
      支持删除未被智能体引用的模型。  
      - 前置校验：仅当无智能体引用该模型时允许删除；若存在引用，则禁止删除并提示原因（如“该模型已被 N 个智能体使用，无法删除”），可引导至引用关系查看。  
      - 二次确认：删除前通过弹窗进行二次确认，避免误删。  
      - 结果反馈：删除成功后列表刷新或移除该行，并给予明确成功提示。  
  - **职责边界说明**：模型管理仅负责模型目录与可用性信息，不承担温度、top_p、最大输出长度等推理参数配置能力，这些参数在智能体配置中设置。

- 支持模型接入可用性验证（连通性与基础响应）  
  - **一键测试入口**：基于模型记录中的后端配置（如模型标识、路由信息等），在模型列表中提供“一键测试”能力，验证是否能成功发起请求并获得基础响应。  
  - **结果反馈与落库**：展示测试是否成功（成功 / 失败）、请求耗时、错误码与错误信息（如超时、认证失败、配额受限等），并将结果状态和必要信息写回模型配置表中的相关字段。
  - **可用性状态标记**：在模型列表中基于模型表中的“最近一次可用性测试结果状态”字段展示“可用 / 存在问题 / 未测试”三种状态，为智能体配置与排查提供参考。

### 4.2 智能体搭建与配置

- 支持智能体基础资料管理（名称、描述）。
- 支持提示词模板化配置，便于场景复用和规范管理。
- 支持配置输出格式要求（结构化或文本化）。
- 支持能力开关配置（如工具调用开关、知识能力开关）。
- 支持智能体维度的模型与推理参数配置  
  - **模型选择**：在智能体配置阶段，从模型管理提供的“启用”状态模型列表中选择一个模型，作为该智能体的运行模型。  
  - **推理参数配置**：选择模型后，在智能体配置界面为该智能体配置温度（temperature）、top_p、最大输出长度（max_tokens）等推理参数。  
  - **默认值策略**：平台为推理参数提供预置默认值，用户可按业务需求在智能体维度调整；  
  - **生效范围**：推理参数与智能体绑定，不同智能体即使使用同一个模型，也可以拥有各自独立的参数配置。

### 4.3 调试与发布管理

- 支持草稿态调试，不影响已发布版本。
- 支持版本管理（草稿、已发布、历史版本）。
- 支持发布前基础校验（配置完整性、关键项检查）。
- 支持发布后快速回滚。

### 4.4 运行观测与成本视图

- 支持查看智能体调用趋势、成功率、异常率等基础指标。
- 支持查看会话级问题定位信息（如失败原因分类）。
- 支持按智能体查看资源消耗与成本趋势，支持运营分析。

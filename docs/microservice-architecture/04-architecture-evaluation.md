# 微服务架构优化评估报告

> 版本：v1.0
> 评估日期：2026-03-19
> 评估范围：`admin-service` / `user-service` / `log-service` / Nginx 网关 / 三份微服务文档
> 评估前提：项目处于早期开发阶段，可清空历史测试数据，从零重构；本报告不讨论迁移平滑性，只讨论方案本身是否合理、是否有遗漏、是否足够指导后续改造

---

## 1. 总体判断

当前三份文档的方向总体正确，尤其是下面三点判断是成立的：

- 把 `admin-service` 从“大后台杂糅服务”重构为更清晰的身份与权限服务，是合理方向。
- 把日志读写职责统一收敛到独立服务，是合理方向。
- 对内部接口增加代码级鉴权、对重复公共代码做统一整理，也是合理方向。

但当前文档还存在四类核心缺陷：

1. **目标服务边界仍不够干净**：`iam-service` 比 `admin-service` 更合理，但仍然聚合了过多领域。
2. **若干设计属于“直接给方案”，但缺少边界原则**：比如为什么要抽 `common-lib`、为什么要上 `Feign + Resilience4j`、为什么日志查询一定迁到 `audit-service`，文档给了结论，但论证还不够。
3. **有些问题诊断不够准确**：部分现状描述和当前代码不完全一致，会削弱文档权威性。
4. **缺少几个真正能指导重构的总表**：服务职责矩阵、数据所有权矩阵、接口/事件边界矩阵。

结论一句话概括：

> 这套改造方案“方向对”，但“架构原则还没写透”，导致它更像一份强执行性的重命名/重组方案，而不是一份高指导性的微服务设计文档。

---

## 2. 当前文档最需要补强的地方

### P0. `iam-service` 的边界仍然偏宽，文档没有正面解释为什么这样聚合

`02-target-architecture.md` 中的 `iam-service` 同时承载：

- auth
- rbac
- account
- platform
- tenant

这比现在的 `admin-service` 已经清晰很多，但它本质上仍然是一个聚合服务，而不是严格意义上的“单一领域服务”。

问题不在于它一定错，而在于文档没有明确回答：

- 为什么 `tenant` 放在 `iam-service` 内，而不是独立成 `tenant-service`
- 为什么 `platform` 治理与 `auth/rbac` 归在一起
- 这是最终边界，还是第一阶段的聚合边界

如果不写清楚，后续读者会误以为：

- “改名成 `iam-service` 以后，边界问题就解决了”

但实际上只是从“命名错误的大后台”变成了“命名更合理的聚合服务”。

建议补充：

- 在目标架构文档中单独新增“为什么 `iam-service` 现在仍然聚合这 5 个模块”
- 明确写出：这是**第一阶段聚合边界**，不是无限制扩张的许可
- 明确未来拆分阈值，例如：
  - 当租户治理开始出现套餐、配额、生命周期、审批流时，再考虑拆 `tenant-service`
  - 当平台治理出现更多平台级运营能力时，再考虑拆 `platform-service`

### P0. 缺少“服务职责矩阵”，导致边界判断仍靠文字描述

现在文档虽然分别描述了三个服务的职责，但没有一张真正清晰的矩阵来回答：

- 哪个服务负责写哪个领域数据
- 哪个服务只读哪个领域数据
- 哪个服务绝不能直接拥有某类数据

建议在文档中补一张类似这样的表：

| 领域 | 主责服务 | 是否允许其他服务直接写 | 跨服务访问方式 |
|------|----------|------------------------|----------------|
| 平台账号 | iam-service | 否 | API |
| 平台角色/权限 | iam-service | 否 | API |
| 租户元数据 | iam-service | 否 | API |
| 租户成员关系 | user-service / 或明确归属 | 否 | API |
| C 端用户画像 | user-service | 否 | API |
| 审计日志 | audit-service | 否 | Kafka + Query API |

没有这张表，后续开发中很容易再次回到“哪个服务方便就往哪个服务里写”的状态。

### P0. 文档没有把“租户成员关系”这个关键边界讲透

当前方案里：

- `iam-service` 管租户、租户角色、租户权限
- `user-service` 管 `tenant_user` 成员关系

这其实是整个系统里最容易重新耦合的边界之一。

现在文档没有讲清楚两个问题：

1. **谁拥有“用户属于哪个租户”这件事的最终事实来源**
2. **谁拥有“用户在租户里具有什么角色”这件事的最终事实来源**

如果这两个问题不明确，后面就会出现：

- `iam-service` 和 `user-service` 都在维护租户相关关系
- 登录判断、成员管理、角色分配分别依赖不同表
- “是成员但没角色”“有角色但成员关系异常”这类边界情况越来越多

建议补充：

- 明确：
  - `tenant_user` 是“成员资格”的唯一来源
  - `tenant_user_role` 是“角色绑定”的唯一来源
- 明确登录、鉴权、成员管理、租户切换分别依赖哪一类事实
- 在架构文档中把这个关系画成一张简图，而不是只放在代码实现细节里

## 3. 具体设计项的评估意见

### 3.1 `common-lib` 值得做，但不能把“相似代码”都收进去

当前文档想把这些内容放进 `common-lib`：

- `Result`
- `TenantContext`
- `BusinessException`
- `BaseEntity`
- `InternalAuthFilter`
- `JwtValidator`
- `IpUtil`

这个方向没有问题，但当前文档的标准还不够严格。

更合适的判断标准应该是：

- **契约性公共件**：适合进 `common-lib`
  - `Result`
  - `InternalAuthFilter`
  - `JwtValidator`
  - 安全相关常量
- **框架耦合型实现**：需要谨慎
  - `BaseEntity`
  - `TenantContext`
  - `BusinessException`

原因不是“以后可能异构语言”，而是更现实的两点：

1. 这些类会把服务内部实现细节变成“共享基础设施”
2. 一旦进入公共库，后续小改动都会变成跨服务同步变更

建议补充：

- 在文档中给 `common-lib` 增加准入规则：
  - 只有“多个服务必须共享且语义稳定”的内容才能进入
  - “只是长得像”的代码不能因为重复就进入

### 3.2 `Feign + Resilience4j` 不是不能用，但文档应先回答“为什么必须用”

当前文档直接把目标方案定成：

- `RestTemplate` → `OpenFeign`
- 引入 `Resilience4j`

这不是错，但文档现在少了一层论证：

- 当前只有少量内部调用，为什么不是“统一 internal client + timeout + shared secret”就够了？
- 为什么这里需要完整引入 `Feign + 熔断`，而不是先做一层轻量封装？

如果不补充这层论证，这部分会显得像“技术栈升级清单”，而不是架构决策。

建议补充：

- 新增一小节“为什么不是继续使用封装后的 RestTemplate”
- 明确采用 Feign 的核心收益：
  - 契约清晰
  - 调用点收敛
  - header 注入统一
  - 测试更易替换
- 明确 `Resilience4j` 解决的是哪类问题，而不是默认所有内部调用都要熔断

### 3.3 `audit-service` 方向合理，但“鉴权模型”还没设计完整

把日志查询 API 迁到 `audit-service` 是对的，因为这能真正让日志域闭环。

但当前文档还缺少一层关键说明：

- `audit-service` 只是“验证 JWT 即可”吗？
- 它是否需要理解平台端/租户端差异？
- 平台日志和租户日志的查询约束由谁强制注入？
- `tenantId` 是由调用方传，还是由 token 强制覆盖？

尤其对于：

- `/api/platform/logs`
- `/api/tenant/logs`

这两个接口，文档需要明确：

- 平台端查询条件哪些允许外部传入
- 租户端查询条件哪些必须被服务端覆盖

建议补充：

- 加一节“audit-service 的认证与授权模型”
- 明确：
  - 平台端：按平台权限控制
  - 租户端：`tenantId` 只能来自 token，不允许前端传入决定

### 3.4 JWT 契约需要在当前范围内统一，但不需要扩展到其他服务

你现在明确只评估当前三服务范围，所以这里不需要写“项目级、跨语言、未来 API Key”之类内容。

这部分更应该收敛成：

- `iam-service` 签发什么 claims
- `user-service` 需要消费哪些 claims
- `audit-service` 需要消费哪些 claims

建议补充一张精简表：

| claim | 含义 | 由谁写入 | 谁会使用 |
|------|------|----------|----------|
| `userId` | 用户 ID | iam-service | user/audit |
| `isPlatform` | 平台端还是租户端 | iam-service | user/audit |
| `tenantId` | 当前租户 ID | iam-service | user/audit |
| `authorities` | 权限列表 | iam-service | audit |
| `tokenVersion` / `tenantVersion` | 失效控制 | iam-service | audit/user |

重点是把当前三服务统一，而不是把范围再次扩出去。

## 4. 当前诊断文档中需要修正的地方

以下内容建议继续保留在评估文档中，因为它们会直接影响原文档的可信度：

- `admin-service/...` 路径应改成 `services/admin-service/...`
- `com.starry.*` 包名表述与当前代码不一致，应修正为现状或明确说明这是目标命名
- `DbFixController` 当前路径是 `/api/db`，不是 `/api/debug/db`
- `PermissionController` 当前走的是 `/api/platform/permissions`，不能和 `/api/rbac/*` 混写
- `SysLogDocument` 当前已核实为两处定义，不是三处

这里建议你在文档里新增一个固定小节：

## 文档校正项

把所有“诊断正确，但现状细节写错”的点集中列出来，避免它们散落在正文里影响阅读。

## 5. 这份评估文档还建议补的两张表

如果你想让这套文档真正具备“高度指导作用”，我最建议再补这两张表。

### 5.1 服务职责矩阵

回答：

- 每个服务负责什么
- 不负责什么
- 和其他服务的边界线在哪里

### 5.2 数据所有权矩阵

回答：

- 哪类表/索引/topic 归哪个服务
- 哪些只能 API 访问
- 哪些可以事件驱动同步

这两张表一旦补上，整个文档的指导性会明显提升。

---

## 6. 精简后的结论

如果只看当前文档覆盖范围，我的建议可以收敛成 5 句：

1. `admin-service` 改成 `iam-service` 是对的，但必须承认它仍是阶段性聚合服务，不是天然干净的最终边界。
2. `audit-service` 独立出来是对的，但必须把查询鉴权模型写清楚。
3. `user-service` 的职责基本合理，但“成员资格”和“租户角色”边界必须明确。
4. `common-lib` 可以做，但只能收真正稳定的契约与安全组件，不能变成共享实现大杂烩。
5. 文档现在最缺的不是更多代码级操作步骤，而是两张总表：服务职责矩阵、数据所有权矩阵。
